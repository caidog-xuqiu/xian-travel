# 数据治理规划（Data Governance）

## 目标
为开放候选空间提供可持续的数据质量底座，降低噪声点、脏点、重复点和过期信息对规划结果的污染。

## 治理对象
- POI（景点）
- 商家（餐饮/服务）
- 标签（适老人/情侣/朋友、室内外、风格标签等）
- 营业信息（开闭店、时段适配）
- 评分/评论（来源可信度与时间新鲜度）

## 核心治理维度
1. 去重（Dedup）
- 名称近似 + 坐标近似 + 类别一致的重复合并
- 连锁门店防泛滥

2. 来源优先级（Source Priority）
- 明确不同来源可信度等级
- 冲突字段按来源优先级决策

3. 新鲜度（Freshness）
- 记录更新时间
- 过期信息降权或隔离

4. 可信度（Trustworthiness）
- 缺关键字段降权
- 异常值（坐标异常、名称异常）隔离

5. 脏数据隔离（Quarantine）
- 明确不可用数据池
- 保留排查依据和原因标签

## 当前阶段实现
- `app/services/data_quality.py` 已进入主链前置位置（discovery 前）：
  - 字段完整性检查
  - 基础去重
  - 来源优先级占位
  - 新鲜度/可信度占位
  - 脏数据隔离与原因标记
- 当前输出最小质量报告：
  - `total_input`
  - `total_after_dedup`
  - `quarantined_count`
  - `issue_counts`
  - `source_distribution`
  - `quality_notes`
- 若治理后可用池过小，会记录质量回退原因，并允许 discovery 回退旧路径，保证主链可用。
- `candidate_discovery` 当前已是多来源输入（existing + local_extended），治理层继续作为发现前护栏，避免扩源后脏数据直接进入候选主链。

## 后续落地建议
1. 扩展质量规则（名称噪声、地理异常、品牌泛滥等）并版本化。
2. 将质量信号更深接入评估体系，持续观测治理收益。
3. 增加“治理规则版本号”，避免缓存与规则漂移冲突。
